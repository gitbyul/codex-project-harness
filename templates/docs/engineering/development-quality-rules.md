# Development Quality Rules

이 문서는 작업 단위마다 지켜야 하는 개발 품질 기준을 정의한다. 하네스의 목표는 특정 언어나 프레임워크를 강제하는 것이 아니라, 모든 작업이 동일한 품질 게이트를 통과한 뒤 커밋과 main 병합으로 이어지게 만드는 것이다.

## 핵심 원칙

- 작업은 작게 나누고 각 작업 단위마다 실행 계획, 검증, 커밋, main 병합, branch/worktree 정리를 완료한다.
- `main`/`master`에서 직접 작업하지 않는다.
- 커밋은 하네스의 상위 완료 명령을 통해서만 수행한다.
- 검증 명령은 `./scripts/verify.sh`로 수렴시킨다.
- 테스트는 빠른 small test, 실제 통합을 보는 medium test, 핵심 smoke 중심 large test로 나눈다.
- 새 결함을 수정할 때는 가능하면 해당 결함을 재현하는 낮은 계층 테스트를 추가한다.
- Mock/Stub/Fake는 외부 경계 통제와 실패 시나리오 재현에만 사용한다.

## 작업 단위 기준

작업 단위는 다음 조건을 만족해야 한다.

- 하나의 명확한 사용자 가치 또는 기술적 변경을 가진다.
- 영향 파일 범위가 실행 계획에 기록되어 있다.
- 검증 방법이 실행 계획에 기록되어 있다.
- QA 계획과 QA 결과가 실행 계획에 기록되어 있다.
- 완료 시 run artifact에 검증 결과와 남은 리스크가 기록되어 있다.

## 품질 게이트

프로젝트별 품질 명령은 `.codex-harness.yml`의 `quality.commands`에 선언한다.

```yaml
quality:
  enabled: true
  commands:
    - npm test
    - npm run test:integration
  required_plan_sections: true
```

명령이 비어 있으면 하네스는 품질 게이트를 건너뛴다. 명령이 있으면 `./scripts/verify.sh` 실행 중 순서대로 실행하며, 하나라도 실패하면 작업 완료와 main 병합이 중단된다.

## 금지 패턴

- 검증 없이 커밋 또는 병합
- 실패한 검증을 이유 기록 없이 우회
- E2E만 추가하고 낮은 계층 테스트를 생략
- flaky test를 retry로만 덮기
- 실행 계획에 QA 계획 없이 작업 진행
- run artifact에 미검증 영역을 남기지 않음
- Mock/Fixture 변경만으로 기능 완료 처리

## 리뷰 체크리스트

- [ ] 작업 단위가 충분히 작은가?
- [ ] 실행 계획에 영향 파일과 검증 방법이 기록되어 있는가?
- [ ] QA 계획과 QA 결과가 기록되어 있는가?
- [ ] `./scripts/verify.sh`가 통과했는가?
- [ ] 품질 게이트 실패를 우회하지 않았는가?
- [ ] 남은 리스크와 미검증 영역이 명시되어 있는가?
