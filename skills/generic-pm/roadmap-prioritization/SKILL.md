# Roadmap Prioritization

Use this skill when deciding what should be built now, later, or not at all. Make prioritization criteria explicit so tradeoffs are reviewable.

## Workflow

1. Read the candidate items and source context: roadmap, PRD, requirements, discovery notes, stakeholder input, constraints, and known risks.
2. Normalize each item into:
   - user or customer value
   - business or product outcome
   - effort or complexity
   - confidence
   - dependencies
   - risk or compliance impact
3. Choose a prioritization approach that fits the input:
   - RICE when reach, impact, confidence, and effort are available
   - MoSCoW when the goal is release scope negotiation
   - phased roadmap when dependencies and sequencing dominate
   - simple decision table when evidence is incomplete
4. Separate:
   - must-have MVP requirements
   - should-have near-term improvements
   - later roadmap candidates
   - explicit non-goals
5. Explain tradeoffs and unresolved assumptions.

## Output Rules

- Write in the language of the target document.
- Do not fabricate scores or evidence. Use TBD or qualitative confidence when data is missing.
- Include a short recommendation and the rationale behind it.
- Preserve hard constraints before scoring optional value.
- Treat documented project constraints as blocking constraints before scoring optional roadmap value.
