# QA Test Strategy

이 문서는 하네스 기반 프로젝트의 테스트 계층과 QA 운영 방식을 정의한다. QA는 마지막 수동 확인 단계가 아니라, 작업 시작 시 실행 계획에 포함되고 작업 종료 시 run artifact에 결과가 남아야 한다.

## 테스트 계층

### Small Tests

- 빠르고 deterministic해야 한다.
- 외부 네트워크, 실제 파일 스토리지, 메시지 큐, 외부 API에 의존하지 않는다.
- 도메인 로직과 작은 단위의 계산/상태 전이를 검증한다.
- 작업 단위마다 가장 자주 실행되는 테스트다.

### Medium Tests

- DB, repository, API route, transaction, contract validation처럼 실제 통합을 검증한다.
- 외부 SaaS나 third-party API는 Fake/Stub 처리할 수 있다.
- main 병합 전 검증에 포함하는 것을 권장한다.

### Large Tests

- 브라우저 E2E, 전체 시스템 플로우, 배포 환경 smoke test를 포함한다.
- 수를 제한하고 핵심 사용자 플로우 중심으로 유지한다.
- 실패 원인이 명확하지 않은 flaky test는 격리하고 수정 계획을 남긴다.

## QA 계획 작성 기준

실행 계획에는 다음 항목이 있어야 한다.

- 수행할 테스트 계층
- 수동 QA 필요 여부
- 회귀 위험
- 검증할 사용자 플로우
- 미검증 영역

## QA 결과 기록 기준

run artifact에는 다음 항목이 남아야 한다.

- 실행한 명령
- 통과/실패 결과
- 수동 확인 범위
- 미검증 영역
- 남은 리스크

## 결함 대응 기준

- 결함 수정 PR에는 재현 테스트를 추가한다.
- E2E에서 발견한 결함은 가능하면 small 또는 medium test로 내려서 재발을 막는다.
- flaky test는 방치하지 않는다. 격리, 수정, 제거 중 하나를 선택하고 이유를 기록한다.

## 리뷰 체크리스트

- [ ] 변경 위험에 맞는 테스트 계층을 선택했는가?
- [ ] DB/API/contract 변경은 실제 통합 테스트로 검증했는가?
- [ ] E2E는 핵심 smoke 중심으로 제한했는가?
- [ ] 수동 QA 필요 여부와 미검증 영역을 기록했는가?
- [ ] 결함 수정에 회귀 테스트가 포함되었는가?
